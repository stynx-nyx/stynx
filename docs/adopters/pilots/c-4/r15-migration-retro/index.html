C-4 Pilot — R15 migration retro

Closed: 2026-06-05. Round: R15 — DEVAI Constitution 0.2.0 migration. Triggering proposal: internal round-tracking note (not published). Predecessor commit (stynx): f54a7f51 (round-open). Migration ADR: ADR-DEVAI-0.2.0-MIGRATION.

TL;DR​

STYNX migrated from DEVAI Constitution 0.1.1 to 0.2.0 in eight waves (W01–W08), executed sequentially over ~2h of wall-clock time, no commits required mid-round. The migration is mechanically successful — f1-paths-present doctor check passes, the published docs IA shifts from one autogenerated category to the canonical seven-section layout, both 0.1.1 and 0.2.0 are snapshotted in versioned-docs, and check-docs-governance flips from 11pass/3fail to 14pass/0fail.

The round surfaced eight upstream-DEVAI gaps worth filing, three wave-prompt vs canonical-shape divergences the operator should know about for future adopter rounds, and one wave-boundary leak where W03 had to absorb W05's typecheck-blocking work and W06 had to absorb W07's stub authoring. The pilot value of STYNX as the C-4 reference adopter is concentrated in those findings — future adopters (PEC, SGP, TEAT) get a much smoother run if DEVAI lands the gaps.

What worked (no friction)​

  • DEVAI's r14-*.mjs script chain (path-rewrite + three fixup passes) was the workhorse of W03. Idempotent, well-scoped to .md files, produced a clear per-mapping reference count. The four-script sequencing matched DEVAI's R14 W03 exactly.
  • r14-harness-sweep.mjs in W05 cleanly rewrote ~630 references across DEVAI's 11 path mappings, including the heavy lifting in packages-web/sdk/src/generated/**/*.ts (auto-generated SDK code embedding OpenAPI-derived paths).
  • The five-step versioned-docs recovery procedure (Plan.md R-4) was the most error-prone single step in R15 and worked exactly as documented in the proposal. git worktree add HEAD → sync there → cp as version-0.1.1 → fresh sync of current → cp as version-0.2.0 → verify versions.json reads ["0.2.0", "0.1.1"]. The "trap" was real but the recovery procedure averted it.
  • devai doctor --adopter in W05 was the closing-pass confidence signal. f1-paths-present ok + all the other checks green confirmed Article 6's tool-layer authority enforcement landed cleanly post-reorg. The single fail it reported (docs-ia.sidebar-curated) was the W06 deliverable's prerequisite, not an unexpected gap.
  • The five gen-*.mjs generators ran to completion against STYNX's tree on first invocation. Drop-in copy worked.
  • check-docs-governance rule count (14) matched the proposal's prediction exactly — 9 R13 rules + 5 R14 IA rules.

Upstream-DEVAI gaps surfaced (in priority order)​

G-1 (HIGH) — r14-*.mjs scripts touch .pnpm-store/ and docs/site/.generated/​

The four r14 scripts' EXCLUDE_DIRS set covers .git, node_modules, coverage, docs/site/build, docs/site/.docusaurus, docs/site/node_modules, docs/site/versioned_docs. It does not exclude .pnpm-store/ (pnpm's content-addressed package cache) or docs/site/.generated/ (the sync output sink which is gitignored on STYNX's setup).

The damage:

  • .pnpm-store/v3/.../undici@7.25.0/index.json got rewritten by W03's stynx-specific rewriter. The file's content-integrity hash now mismatches pnpm's expected value; pnpm refetches on the next pnpm install and re-stores. Harmless in practice but it taints the cache.
  • r14-fixup-relpath-3.mjs touched 1393 files of which 1212 were in docs/site/.generated/. The script reports "1393 files touched" which gives an alarming headline despite ~87% being regeneratable cache.

Filing: add .pnpm-store/ and docs/site/.generated/ to all four r14-*.mjs EXCLUDE_DIRS sets, and to r14-harness-sweep.mjs likewise. Adopter rounds that follow STYNX inherit the same .pnpm-store/ issue.

G-2 (HIGH) — r14-*.mjs scope is .md only; misses .json and .ts/.mjs/.cjs path strings​

The path-rewrite script in W03 produced clean .md output but spec-validate-all immediately surfaced 8 invariant-file errors: every docs/framework/arch/invariants/INV-*.json carries a "doc": "docs/&lt;old&gt;/X.md" field that the script left untouched. Similarly docs/framework/arch/trace.json (20 refs), docs/framework/contracts/errors.json (1 ref), docs/framework/schemas/error-envelope.schema.json description (1 ref).

pnpm typecheck separately revealed 4 OpenAPI-consuming script files (scripts/verify-api-coverage.mjs, packages/core/scripts/emit-openapi.mjs, scripts/verify-release-provenance.mjs, scripts/verify-production-readiness-reference.mjs) plus tools/migration-linter/test/migration-linter.spec.ts and the sync-content script's URL substitutions.

Filing: extend the script chain (or add a fifth script) to cover JSON "doc": / "openapiPath": fields and TS/JS string-literal path references. The r14-harness-sweep.mjs already covers strings in TS/JS but with a different scope; its MAPPING is what should be borrowed for the JSON-aware variant.

G-3 (HIGH) — Generator portability: 5 gen-*.mjs files emit DEVAI-framework-relative URLs​

Each of the 5 generators produces markdown referencing DEVAI's own framework prose (constitution.md, scorecard.md, substrates.md, transversals.md, test-policy.md, meta/adr/ADR-DOCS-IA.md). For DEVAI itself these are intra-repo; for any adopter using Decision A's hybrid (the prescribed library shape), they break.

The post-W07 docs-links count is 218; +24 vs the W06 baseline of 194 are entirely generator-output → DEVAI-framework references.

Filing: add --framework-base-url &lt;url&gt; parameter (or read from categories.json) to all five generators. The default at DEVAI itself stays as relative (preserving today's behavior); adopters pass --framework-base-url https://aarusso-nyx.github.io/devai/docs/framework/ (or similar) and the generators rewrite cross-framework references as absolute.

G-4 (MEDIUM) — Pack-override surface for docs-ia.* severity is forward-looking but not implemented​

The proposal's Decision D promised adopter pack-overrides to graduate the 5 new docs-ia.* rules from default warn to fail. Inspection of external DEVAI sibling checkout reference (not published) shows severities are hardcoded per rule with no override surface — the proposal's promised mechanism doesn't exist yet.

Empirically irrelevant for R15 because all 5 rules already return pass against STYNX's post-W06 state, so there's nothing to graduate. But future adopters who hit a docs-ia rule failure will discover they cannot graduate or downgrade per-pack.

Filing: implement the per-rule severity override surface DEVAI's ADR-DOCS-IA Decision 9 promised. Adopter pack configs at .devai/config/packs/&lt;adopter&gt;.json (or similar) carry rules: &#123;&lt;ruleId&gt;: &#123;severity: "fail"|"warn"|"pass"&#125;&#125; and the sensor applies them after the default-severity hardcode.

G-5 (MEDIUM) — Generator default output path assumes DEVAI's monorepo​

DEVAI's generators default to writing under docs/site/docs/... (matching path: 'docs' in DEVAI's Docusaurus config). STYNX reads from docs/site/.generated/site-docs/ (per the R14 sync substrate). When W07 first ran the generators, output landed in docs/site/docs/ — a path that didn't exist pre-W07, that Docusaurus doesn't read, and that has no business existing in STYNX's tree.

Filing: default output should be either (a) categories.json-driven, reading outDir per-section, or (b) explicit-required (no default, all five take --out / --out-dir mandatory). Adopters with a custom Docusaurus path (any non-DEVAI layout) silently produce orphan output otherwise.

G-6 (MEDIUM) — docs-links sensor counts .generated/ brokens (double-counting)​

The W06 sync output expanded .generated/site-docs/ from sparse to full. Every broken markdown link in source content now appears twice — once in docs/&lt;path&gt;/file.md and once in docs/site/.generated/site-docs/&lt;path&gt;/file.md. The W06 → W07 delta of +13 (W05→W06) and the W07 → W08 deltas include sync mirrors of source brokens.

The proposal's eight-wave compressed plan didn't anticipate this. STYNX's pre-R15 baseline was 181; post-R15 source-side count is similar but sensor count is ~218.

Filing: add docs/site/.generated/ (or whichever path matches the project's output_dir config) to docs-links sensor's default EXCLUDE_DIRS.

G-7 (LOW) — check-adrs hardcodes docs/meta/adr/ post-0.2.0​

This works for STYNX (the W03 reorg moves ADRs there) but the sensor's path string is fixed in code. If an adopter elects to keep ADRs at a different location (e.g. inherits from their own pre-existing structure), they have to either rewrite the sensor or move their ADRs.

Filing: check-adrs should read its target path from categories.json (the manifest already declares meta/adr is the ADR home for the manifest's owner; the sensor can read the same source).

G-8 (LOW) — gen-skill-catalog shells out to devai skill list​

The generator runs devai skill list and parses output. For an adopter without devai on PATH (or running the sibling CLI via node ../devai/packages/cli/dist/bin.js), the catalog generator may error or produce empty output. STYNX's run produced 1 page (empty placeholder) without erroring, but this is fragile.

Filing: generator should either accept skill-list as input (--from &lt;path&gt;) or detect the CLI path automatically (look at categories.json for the adopter's CLI invocation).

Wave-prompt vs canonical-shape divergences​

Three places where the W01-authored wave prompts disagreed with what DEVAI's R14 actually delivered. In each case the worker followed canon over prompt and documented the deviation in the wave log.

D-1 — W03 prompt placed adr/ops/security under framework/; canon places them under meta/​

The W03 prompt's step A enumerated 17 git mv operations including docs/&#123;adr,ops,security&#125; → docs/framework/&#123;adr,ops,security&#125;/. Inspecting DEVAI's R14 output (the canonical 0.2.0 layout) and the r14-path-rewrite.mjs ABS_MAPPING (the authoritative migration source) revealed all three live at docs/meta/&#123;adr,ops,security&#125;/. Worker corrected.

Implication for future adopter prompts: destinations should be derived from the script's MAPPING, not re-prosed in the prompt. Prose drift is silent.

D-2 — W04 prompt's categories.json shape was wrong​

The W04 prompt described a manifest with sections[*].subdirs[] arrays. DEVAI's actual docs/_ia/categories.json (the R14 W05 deliverable, what sync-content.mjs consumes) is sections + categories + rootFileAllowlist + outOfIa. Worker followed canon.

Implication: prompt authors should cat ../../devai/docs/_ia/categories.json and mirror the shape verbatim, not invent.

D-3 — W06 prompt used CLI-snapshot procedure that's unreliable in Docusaurus 3.x​

The W06 prompt called npx docusaurus docs:version 0.1.1 against the current .generated/site-docs/ to snapshot 0.1.1. That snapshots the CURRENT (0.2.0) content as the 0.1.1 version — wrong content. The actual procedure required a git worktree + sync-on-worktree to materialise the OLD content first, then copy that to versioned_docs/version-0.1.1/.

The five-step recovery procedure documented in Plan.md R-4 (lifted from DEVAI R14 W05's actual fix at commit b700657) IS the right procedure. The W06 prompt's commands were a partial restatement that didn't preserve the "OLD content first" step.

Implication: R-4 procedure should be the prompt body verbatim, not abbreviated.

Wave-boundary leaks​

L-1 — W03 absorbed W05's typecheck-blocking harness paths​

The r14-path-rewrite.mjs script handles .md files. pnpm typecheck failed on 4 OpenAPI consumer paths + 1 test fixture + various script files that the script didn't see. To clear W03's "mandatory minimum gates pass" acceptance criterion, the worker had to do the W05 sweep early on those specific files. W05's three-pass discipline was still run later (and caught 4 additional grep hits) but the chunk that blocked typecheck got absorbed into W03.

Implication: the wave-boundary rule (W05 = engineer + inspector, W03 = engineer) is logical but the mandatory-gate rule supersedes wave boundaries. Future adopter prompts should either (a) acknowledge this absorption explicitly in W03 OR (b) declare the typecheck gate as W05-deferred.

L-2 — W06 absorbed W07's theory/ and roles/ stubs​

Docusaurus 3.x's empty-category refusal means the seven-section sidebar can't build without content under every section. start/, theory/, roles/ were empty (only start/index.md from W03's loose-file rename). W07's prompt called for §1 Start + §4 Roles authoring — but those had to happen in W06 to clear the build gate.

Implication: the W06 build verification is what forces theory/roles stubs to W06. Future prompts should declare this absorption.

Findings unique to STYNX as a library adopter​

  • Source code consumers of .devai/config/project.json: zero. The pin-bump baseline-shape risk (Plan.md R-1) was theoretical; STYNX is data the harness operates on, not a harness consumer. Adopters with no vendored DEVAI CLI inherit this insulation.
  • STYNX has no F1 substrate slices (invariants/journeys/trace/glossary files_scanned: 0 for the four spec validators when STYNX's invariant files aren't found in the validator's scope path). docs/framework/arch/invariants/*.json exists post-W03 but the validator finds 0 files until after the path-fixup; then 9 invariants surface.

Decisions A–D — final disposition​

  • Decision A (§3 hybrid Framework): landed as authored. docs/framework/index.md carries the §3.0 cross-link + STYNX's local contract surface enumeration. Detailed prose for sub-sections deferred per ADR.
  • Decision B (stynx-specific dir mapping): landed; W04's categories.json covers all subdirs including the 5 the proposal audit missed. docs/stynx/ resolved to docs/adopters/stynx/.
  • Decision C (loose-root files kebab-case): landed at move time in W03.
  • Decision D (severity graduation to fail): ASPIRATIONAL — the pack-override surface doesn't exist in DEVAI yet (see G-4). Functionally irrelevant for R15 close because all 5 rules already pass. The decision-to-graduate stands; the mechanism to execute it doesn't yet.

STYNX-side TODO carried forward​

  1. Cleanup the 181→218 pre-existing brokens in a future content-cleanup round. Includes the navbar/footer refinement to fully tighten onBrokenLinks back to 'throw'.
  2. §3 Framework deep fill — Decision A explicitly defers detailed prose for framework/&#123;arch,contracts,api,schemas,product,glossary&#125;/ to a future round.
  3. Working-notes split — docs/adopters/stynx/working-notes/ could move to meta/ for cleaner role-substrate alignment. Single-move-pragmatism deferred the split.
  4. Generator contract tests — adopt DEVAI's R14 W10 vitest fixtures for the 5 gen-*.mjs scripts to catch silent output drift.
  5. Round-artifact location harmonisation — STYNX uses internal round-tracking note (not published) (outside the repo) where the canon assumes internal work note (not published) (inside). The .gitignore log exception added in W01 is structurally a no-op. Move or document.

Recommendation for PEC / SGP / TEAT (next adopters)​

If you are migrating to constitution 0.2.0:

  1. Apply G-1 first. Pre-extend the r14-*.mjs scripts with .pnpm-store/ and docs/site/.generated/ excludes before running them. Saves you the cleanup at W03 close.
  2. Apply G-2 second. Either patch the scripts to cover .json "doc": and .ts/.mjs string literals, or budget for a hand-sweep at W05 close.
  3. Use the five-step versioned-docs procedure from Plan.md R-4 verbatim. Do not improvise. The git-worktree + sync-on-worktree dance is the only correct path.
  4. Expect the build to fail until navbar URLs are updated. Budget that for late in your equivalent W07.
  5. Don't expect pack-override severity graduation to work yet (G-4) — keep your docs-ia.* rules at the default warn severity until DEVAI ships the surface.
  6. STYNX's R15 closeout is your reference. This retro + the wave logs at internal round-tracking note (not published) are the operational playbook.

If you are a non-library adopter (application/runtime-host with repo.kind != library): the proposal's Decision A hybrid does not apply to you. Your §3 Framework strategy may differ.

Closing​

R15 closed clean at the round level: all mandatory minimum gates pass, all scope-conditional gates pass, sensor verdict is pass. The publish dry-run succeeded (8227 files, 161 MB). Live publish + pnpm ci:stynx are deferred per the with-blockers disposition for environmental reasons (Docker not running locally; gh-pages push out of session scope).
